# 给 Agent 做体检：一套可操作的评估方法论

你搭了一个 Agent，它能跑了。Demo 演示的时候一切正常，老板很满意。

但你真的知道它跑得好不好吗？

"能跑"和"跑得好"之间，隔着一整套评估体系。没有评估的 Agent，本质上是一个高级随机数生成器——它可能偶尔给出惊艳的答案，也可能在你最需要它的时候掉链子，而你完全无法预知是哪种情况。

这篇文章要解决的就是这个问题：**怎么给你的 Agent 做一次全面体检，而且建立一套可持续的评估机制。**

---

## 一、为什么需要评估

### 没有评估的 Agent 是随机数生成器

传统软件测试很简单：输入 A，预期输出 B，运行检查——等于就是对，不等于就是错。但 Agent 是概率性的。给它同样的输入，它可能产出 B、C 或 D，每次运行的路径都可能不同。

这意味着你不能用"等于/不等于"来判断 Agent 的输出。你需要一套全新的评估思维。

评估的核心是三元组：**输入（Input）+ 输出（Output）+ 评分器（Grader）**。传统测试的评分器就是"等于"，Agent 评估的评分器可以是"代码能跑通""格式正确""覆盖了所有要点"——更灵活，但也更难设计。

### Pass@k 和 Pass^k：两个经常被混淆的指标

衡量 Agent 性能，有两个关键指标：

- **Pass@k**：生成 k 个答案，至少有一个通过就算 Pass。衡量的是**能力上限**——模型在最好情况下能做多好。
- **Pass^k**：生成 k 个答案，全部通过才算 Pass。衡量的是**可靠性下限**——模型在最差情况下不会差到哪去。

```
Pass@5 = 90%  → 5 次尝试内，90% 概率至少答对一次（适合代码生成）
Pass^5 = 70%  → 5 次回答，70% 概率全部正确（适合关键决策）
```

大多数场景关注 Pass@k——Agent 可以尝试多次，只要最终能找到正确答案。但关键场景（金融决策、医疗建议）必须关注 Pass^k——不能容忍任何错误。

### 评估金字塔

评估不是只有一个层级。从底层往上：

```
        用户满意度（最真实，最难收集）
       ╱
    端到端评估（完整流程，成本最高）
   ╱
  集成评估（多步协作）
 ╱
单元评估（单步检查，最快最便宜）
```

**从单元评估开始逐步向上。** 单元评估保证基础质量，集成评估保证协作正确，端到端评估保证整体效果。跳过底层直接做端到端评估，就像不测零件直接组装飞机——能飞，但你敢坐吗？

## 二、四类评估与纵深防御

Agent 评估不是一招鲜吃遍天。不同类型的 Agent，评估策略差异巨大：

| Agent 类型 | 评估难度 | 核心评估维度 |
|:---|:---|:---|
| Coding Agent | ⭐ | 测试通过率、代码质量、安全扫描 |
| 对话 Agent | ⭐⭐ | 准确性、相关性、完整性 |
| 研究 Agent | ⭐⭐⭐ | 覆盖面、来源质量、分析深度 |
| Computer Use Agent | ⭐⭐⭐⭐ | 任务完成率、操作效率、错误恢复 |

但不管你评估什么类型的 Agent，单一评估都有漏洞。这里引入一个概念——**瑞士奶酪模型**：

```
[代码评分器] 全覆盖 → 漏洞 A（判断不了语义）
    ↓
[模型评分器] 抽样 10-20% → 漏洞 B（可能有评分偏见）
    ↓
[人类评分器] 关键节点 → 覆盖漏洞 B
    ↓
[用户反馈] 持续收集 → 最终兜底
```

每层评估都有盲区，像瑞士奶酪叠在一起——一层的洞被另一层盖住，整体防御远比单层可靠。

**纵深防御实践：**

1. **代码评分器（全覆盖）**：格式检查、语法验证、测试执行。每个输出都过，成本近零。
2. **模型评分器（抽样）**：对通过的输出做语义检查。随机抽样 10-20%，深度评估。
3. **人类评分器（关键节点）**：高价值输出的最终验收、评分器校准。
4. **用户反馈（持续）**：真实用户的使用反馈，最真实但有延迟。

## 三、实操：用 8 维度框架评估 Agent 可行性

理论讲完了，来点实操。这里介绍一个我一直在用的评估框架——**Agent 项目可行性评估**，8 个维度，每个维度 1-5 分，满分 40。

这个框架回答的核心问题是：**我该不该做这个 Agent？**

### 8 个维度速览

```yaml
dimensions:
  - name: 输入结构化度
    question: "输入是结构化表单还是自由对话？"
    high: "结构化数据、API 输入"
    low: "模糊意图、创意发散"

  - name: 输出可验证性
    question: "怎么知道 Agent 做对了？"
    high: "代码编译通过、格式校验"
    low: "无客观标准、纯审美判断"

  - name: 失败代价
    question: "Agent 做错了会怎样？"
    high: "无影响，可轻松回滚"
    low: "法律、安全、不可逆操作"

  - name: 上下文可得性
    question: "Agent 需要的信息能不能完整给到？"
    high: "所有信息在数据库中"
    low: "完全依赖个人经验和直觉"

  - name: 迭代价值
    question: "多步推理能显著提升质量吗？"
    high: "本质就是多步任务"
    low: "纯查表，不需要推理"

  - name: 执行频率
    question: "这个任务多久做一次？"
    high: "每天多次"
    low: "一次性或极少"

  - name: 人力成本
    question: "现在用人做要花多少？"
    high: "占用大量人力，枯燥重复"
    low: "几乎不需要人力"

  - name: 领域复杂度  # 反向评分
    question: "需要多少领域知识？"
    high: "通用任务，不需要特定知识"
    low: "依赖专家直觉和多年经验"
```

### 一票否决规则

总分高不等于可以做。以下任一条件满足，直接标记为 ❌：

- **失败代价 = 1 且无有效人工兜底**：不可逆高风险操作，没有人类审核环节
- **上下文可得性 = 1**：核心信息完全无法形式化，Agent 注定"瞎"
- **输出可验证性 = 1 且失败代价 ≤ 2**：无法验证对错，但错误代价高

### 三个真实场景的评估结果

**场景 A：代码 PR 自动审查**

| 维度 | 分数 | 说明 |
|:---|:---|:---|
| 输入结构化度 | 4 | 代码 diff 是半结构化的 |
| 输出可验证性 | 4 | 可检查格式、关键词覆盖 |
| 失败代价 | 4 | 漏掉 bug 不致命，人工终审兜底 |
| 上下文可得性 | 4 | 代码库 + 风格指南可读取 |
| 迭代价值 | 5 | 需要理解上下文、检查依赖影响 |
| 执行频率 | 5 | 每天多次 PR |
| 人力成本 | 4 | code review 占大量工程师时间 |
| 领域复杂度 | 4 | 需要编程知识但可文档化 |
| **总分** | **34/40** | ✅ 立即启动 |

**场景 B：自动生成营销文案**

| 维度 | 分数 | 说明 |
|:---|:---|:---|
| 输入结构化度 | 3 | 产品描述有结构，但品牌调性难约束 |
| 输出可验证性 | 2 | 文案质量主观性强 |
| 失败代价 | 3 | 品牌形象风险，需人工审核 |
| 上下文可得性 | 3 | "调性"部分难以完全形式化 |
| 迭代价值 | 4 | 多轮改写能明显提升质量 |
| 执行频率 | 4 | 每天需要多条文案 |
| 人力成本 | 4 | 营销团队大量时间花在文案上 |
| 领域复杂度 | 3 | 需要营销知识和品牌理解 |
| **总分** | **26/40** | 🟡 有条件启动，先做"草稿 + 人工修改" |

**场景 C：自动处理客户退款（含一票否决）**

| 维度 | 分数 | 说明 |
|:---|:---|:---|
| 输入结构化度 | 4 | 工单格式，有历史订单数据 |
| 输出可验证性 | 4 | 规则明确 |
| **失败代价** | **2** | **错误退款 = 直接财务损失** |
| 上下文可得性 | 4 | 订单数据 + 退款政策可获取 |
| 迭代价值 | 3 | 需要判断退款条件 |
| 执行频率 | 5 | 每天大量请求 |
| 人力成本 | 5 | 客服团队最大痛点 |
| 领域复杂度 | 4 | 退款规则可文档化 |
| **总分** | **31/40** | ❌ 一票否决：失败代价高，需加人工审批 |

看到了吗？场景 C 总分 31，看起来很高，但一票否决规则直接把它拦下了。这就是评估框架的价值——**防止你在高分的幻觉中踩坑**。

## 四、从"能跑"到"跑得好"

评估不是上线前做一次就完事了。它是持续的过程。

### 建立评估基线

第一步是建立基线——用当前版本的 Agent 跑一遍评估集，记录各项指标。后续每次改动，都和基线对比：

```json
{
  "baseline": {
    "date": "2026-03-30",
    "model": "gpt-4o",
    "pass_at_1": 0.72,
    "pass_at_5": 0.91,
    "avg_cost_per_run": 0.034,
    "avg_latency_ms": 2800
  },
  "latest": {
    "date": "2026-04-15",
    "model": "gpt-4o",
    "pass_at_1": 0.78,
    "pass_at_5": 0.93,
    "avg_cost_per_run": 0.031,
    "avg_latency_ms": 2400
  }
}
```

**趋势比绝对值更重要。** 如果 Pass@1 在下降，即使当前数字还行，你也应该警觉。

### 评估驱动开发（Eval-Driven Development）

把评估融入开发流程，而不是事后补：

```
1. 定义你要解决的问题
2. 写 10-20 个测试用例（输入 + 期望输出 + 评分器）
3. 跑一遍，建立基线
4. 改 Prompt / 换模型 / 调参数
5. 跑评估，和基线对比
6. 有提升 → 合入；没提升 → 回滚
7. 定期补充新的失败案例到评估集
```

每次发现 Bug，第一反应不是修 Bug，而是问：**"为什么评估没检测到这个？"** 然后把对应的测试用例加进评估集。这样你的评估体系会越来越强。

### 评估漂移：评估体系本身也要被评估

两个常见问题：

1. **评估饱和**：所有测试通过率都接近 100%，评估从"能力测试"退化为"回归测试"——只能告诉你"没退步"，无法衡量"进步了多少"。
2. **评估偏移**：模型行为变化导致评分器给出不准确的分数。比如评分器期望 `96.1249911`，模型输出 `96.12` 被扣分——实际是评分器太僵化。

**应对策略：** 定期审查评估结果分布，分数方差趋近于零说明评估已失效；每次模型升级后人工抽检 5-10%。

## Take-away

Agent 工程的成熟标志不是"能跑"，是"知道跑得好不好"。

- 没有评估的 Agent 是随机数生成器
- 用 8 维度框架在项目启动前做可行性评估，防止方向性错误
- 纵深防御：代码评分器 → 模型评分器 → 人类评分器 → 用户反馈
- 评估驱动开发：把评估融入开发循环，而不是事后补
- 评估本身也会漂移，需要定期维护

下一次你搭完一个 Agent，先别急着庆祝。打开评估框架，给它做个体检。
