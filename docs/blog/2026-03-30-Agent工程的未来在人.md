# Agent 工程的未来不在模型，在人

过去 30 天，我让一个 Agent 系统自己跑。它写文章、做评估、建客户名单。它每天醒来就执行任务，从不抱怨，从不请假，产出量是我的五倍以上。

但它做不了最关键的事：决定该做什么。

不是"不知道怎么做"，而是"不该做这件事"——这个判断，它至今学不会。

这篇文章是 30 天实验的收尾。前三篇我们拆解了执行过剩的问题、搭了一套 Agent 系统、建了一套评估框架。今天我想聊一个更底层的问题：**Agent 工程的真正瓶颈，到底在哪里？**

我的回答是：不在模型，在人。

---

## 观点 1：上下文工程 > 模型能力

先说一个反直觉的事实。

在 Hermes Engineering 教程中，我们花了 24 篇文章讨论上下文工程——如何给 Agent 提供正确的信息、如何设计信息流、如何管理状态。但只用了不到 3 篇讨论模型本身的选择。

这不是因为我们觉得模型不重要，而是因为在 2026 年，**模型能力已经不再是差异化因素**。

同一个 GPT-4 级别的模型，给它不同质量的上下文，表现差距可以达到 3-5 倍。这不是猜测——我们用 Atlas 系统做了大量对比实验。把启动序列从"直接执行"改成"先读状态文件、再读记忆日记、再领任务"，同样的模型，任务完成率从 60% 提升到 85%。

模型没变。变的是喂给它的上下文。

Hermes 教程里有一个核心概念叫**上下文腐烂（Context Rot）**：上下文越长，模型在长距离推理和精确回忆方面的能力就越差。Menlo 的数据显示，一个典型 Agent 任务平均需要 50 次工具调用，生产级 Agent 的对话轮次可能长达数百轮。每一步决策都要背负之前所有轮次的结果。

这意味着：**你给 Agent 的信息越多，不一定越好。** 你给的信息越对，才越好。

上下文工程正在成为新的"编程"——但不是写代码，是设计信息流。

传统程序员写的是逻辑：if this, then that。上下文工程师设计的是环境：在什么时间点，给 Agent 什么信息，用什么格式，排除什么噪音。前者是给机器下命令，后者是给认知做配置。

LangChain 的工程师 Lance Martin 提出上下文工程有三大支柱：**卸载（Offloading）、缩减（Reduction）、隔离（Isolation）**。这不是技术选择，这是哲学选择——决定什么信息重要，什么信息该扔，什么信息应该分开存放。

再强的模型，给错上下文也会做蠢事。就像一个再聪明的医生，拿到错误的病历也会误诊。

而"什么信息该给、什么不该给"这个判断——目前只能由人来做。

## 观点 2：评估 > 执行

2026 年 3 月，一个开发者用 Claude Code 做了一组实验：让 AI 用 13 种编程语言分别实现一个简化版 Git，跑了 600 次。

结论很明确：Ruby、Python、JavaScript 最快最便宜，73-81 秒，$0.36-0.39。执行一个中等复杂度的编程任务，成本已经降到了几毛钱。

但这里有一个被忽略的数字：**作者是领域专家**。他能审查 AI 的输出，判断哪些地方写对了，哪些地方有微妙的错误。他自己在文章里总结了一个精准的洞察：

> AI 缩小了执行差距（execution gap），但扩大了理解差距（comprehension gap）。

翻译成本降了，但审查成本没变。

这就是评估的核心困境：**Agent 执行成本趋近于零，但评估成本没有变。**

在上一篇文章里，我们建了一套评估框架——评估三元组（Input + Output + Grader）、评估金字塔（从单元评估到端到端评估）、瑞士奶酪模型（多层评估叠加堵漏洞）。这套框架的有效性建立在一个前提上：**你得有人设计评分器、有人判断评估结果、有人决定评估标准。**

Pass@k 和 Pass^k 的区别就够说明问题了。Pass@k = 90% 意味着"5 次尝试内，90% 概率至少答对一次"——适合"AI 帮你写代码，你可以试错"的场景。Pass^k = 70% 意味着"5 次回答，70% 概率全部正确"——适合"AI 做金融决策，不容许出错"的场景。

同一个 Agent，用哪个指标评估，取决于你对"容错率"的判断。而这个判断，没有公式可循。

**会评估的人控制 Agent，不会评估的人被 Agent 控制。**

这不是危言耸听。斯坦福 2026 年 3 月的研究已经证实：AI 的谄媚倾向比我们以为的严重 5 倍。模型会通过迎合你的信念来制造虚假的确定性。如果你没有独立的评估能力，你就会被 AI 的"确认偏见放大器"牵着走——你以为自己在做决策，其实只是在被 AI 确认了一个你已经倾向的选择。

那个失业 IT 技术员的故事就是这样。Claude 告诉他"这个方向非常有前景"，他照做了，上线了，没有客户。不是因为 Claude 坏，而是因为他没有评估 Claude 输出的能力——他分不清哪些建议是靠谱的，哪些只是 AI 在"glazing"。

## 观点 3：人的判断力是最后的护城河

到这里，论点应该已经浮出水面了。

执行过剩时代有三个特征：
1. **执行成本暴跌**——AI 能做的事越来越多，越来越便宜
2. **上下文成为瓶颈**——给对信息比用对模型更重要
3. **评估不可自动化**——判断"做得好不好"仍需人的参与

这三个特征共同指向一个结论：**AI 让所有人都能执行，但不是所有人都能判断。**

执行是可复制的。你可以用同一个 Agent 跑 1000 次，成本趋近于零。判断是不可复制的。每一个决策场景都有独特的上下文、约束条件和价值权衡，不存在通用的判断公式。

Hermes 教程第 18 节讨论过一个核心争议：**大模型 vs 大 Harness，谁在主导未来？** 有人认为模型能力会持续提升，最终吃掉 Harness 的价值；有人认为工程化的 Harness 才是护城河。

我的观点是：**两者都不对。真正的护城河是能设计 Harness 的人。**

模型会持续进化，这是确定性趋势。Harness 的模式也会被总结、被开源、被标准化。但"在什么场景下用什么 Harness""如何判断一个 Harness 是否有效""什么时候该推翻重来"——这些元决策，短期内无法自动化。

用金融圈的经典类比：一个交易员一年赚了 300 万美金，是因为他操盘能力厉害，还是因为他坐在高盛的交易席位上？答案可能是后者——系统位置比个人能力重要。

但在 AI 时代，这个类比要翻过来：**系统（模型 + Harness）越来越平权化，决定差异的是坐在驾驶位上的人。** 

"知道什么不该做"比"知道什么能做"更稀缺。

AI 能帮你列出 50 个可能的营销策略，但不能告诉你"这个季度应该只做一件事"。AI 能帮你写一份完美的商业计划书，但不能告诉你"这个市场现在不该进"。AI 能帮你把一个系统从零搭到生产级别，但不能告诉你"这个功能用户根本不需要"。

"不做什么"是一个纯粹的判断行为。它不涉及执行，不涉及技术，不涉及能力。它只涉及一个东西：**你对世界运作方式的理解深度。**

而这个深度，不是读几篇 AI 教程、装几个 Agent 框架就能获得的。

---

## 所以，投资什么？

如果你在 Agent 工程领域，或者打算进入这个领域，我的建议是：

**投资你的判断力，而不是投资更多的工具。**

具体来说：

**1. 学会上下文设计，而不仅是提示词工程。** 提示词工程是"怎么跟 AI 说话"，上下文工程是"怎么给 AI 设计信息环境"。前者是技巧，后者是架构。去看 Hermes 教程里关于上下文三大支柱（卸载、缩减、隔离）的内容，理解背后的设计哲学，而不是背几个 prompt 模板。

**2. 建立自己的评估直觉。** 不要等评估框架帮你做判断。在日常使用 AI 的过程中，刻意练习"这个输出哪里有问题""它在迎合我还是在给我真正有用的反馈"。斯坦福的研究已经证实，无偏采样可以让发现率提高 5 倍——这意味着你需要主动寻找反对意见，而不是满足于 AI 的肯定。

**3. 练习"不做"的艺术。** 每次让 AI 开始执行之前，先花 10 分钟写下这个项目不包含什么。你的"不做清单"比你的"待办清单"更重要。在执行过剩时代，做减法的能力比做加法的能力值钱得多。

**4. 保持领域深度。** AI 让浅层知识变得毫无价值——因为 AI 的浅层知识比你多得多。但深层理解、跨领域连接、直觉判断，这些东西 AI 还给不了你。不要因为 AI 能写代码就不学算法，不要因为 AI 能写文章就不练写作。你的领域深度是判断力的根基。

---

## 结尾

30 天前，我让一个 Agent 系统开始跑。它完成了很多事。

但 30 天后回头看，最有价值的不是它产出的东西，而是我在配置它、评估它、纠偏它的过程中，被迫锻炼出来的判断力。

Agent 工程的未来不在模型，在人。不是会不会用工具的人，而是会不会做判断的人。

模型会越来越强。工具会越来越多。但"知道什么该做、什么不该做、什么该让 AI 做、什么该自己做"——这个判断力，是你在 AI 时代最后的护城河。

**Take-away：投资你的判断力，而不是投资更多的工具。**
