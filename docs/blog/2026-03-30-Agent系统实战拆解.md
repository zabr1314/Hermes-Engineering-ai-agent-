# 用我自己的 Agent 系统，拆解 Hermes Engineering 教程里的每个模式
## How My Agent System Validated (and Challenged) Every Pattern in the Hermes Engineering Tutorial

我搭了一个 Agent 系统，让它自己跑了一晚上。第二天早上看结果，发现它把我教程里写的每个坑都踩了一遍。

这个系统叫 Atlas，跑在 OpenClaw 上，有一个执行引擎、一个状态管理器、一个记忆压缩器、一个启动序列，还有一堆按需加载的 Skills。它不完美，但它活着，每天自己跑任务、写文章、做分析。而我恰好也在写一套叫 Hermes Engineering 的 Agent 工程教程。

两个项目撞到一起，产生了一种奇妙的化学反应：**教程里的每个理论模式，Atlas 都用真实的代码和踩坑经历给出了注脚。** 有些印证了理论，有些暴露了理论没说到的东西。

这篇文章就是一次拆解——用 Atlas 的实际架构，逐个对照教程里的核心模式。

---

## 1. 启动序列 — 每次醒来都是失忆的
### The Boot Sequence — Waking Up Amnesiac Every Time

**教程说：** 长运行 Agent 每次会话都是一次从零开始。标准步骤是：定位（确认目录）→ 回忆（读进度记录）→ 领任务（读功能清单找未完成条目）→ 复原（启动环境）→ 验证（确认基础功能）→ 开工。

**Atlas 怎么做的：** 我写了一个 `BOOT_SEQUENCE.md`，五步走——确认时间 → 用 `state.py status` 和 `state.py next` 读状态 → 读今日记忆日记 → 领任务 → 按任务类型渐进式加载文件。

```markdown
## 步骤 1：定位
确认当前在哪个上下文中。
## 步骤 2：读状态
python3 ~/.openclaw/workspace/looper/state.py status
python3 ~/.openclaw/workspace/looper/state.py next
## 步骤 3：读今日记忆
cat ~/.openclaw/workspace/memory/$(date +%Y-%m-%d).md
## 步骤 4：领任务
根据 state.py next 的输出，决定本次执行什么。
## 步骤 5：渐进式加载（按需）
不要提前加载所有文件。只加载当前任务需要的。
```

**踩了什么坑：** Atlas 的第一版根本没有启动序列。每次心跳触发时，Agent 靠"印象"工作——上一次说了什么、当前进度到哪了，全靠上下文窗口的记忆。结果很明显：上下文窗口一压缩，Agent 就开始重新发明轮子。它会重新分析我已经分析过的问题，重新定义我已经定义过的服务，甚至试图重新创建已经存在的文件。

**结论：** 启动序列不是"可选的优雅"，是"必须的基础设施"。教程里 Anthropic 的方案写得很清楚——初始化 Agent 只存在于第一天，之后的编码 Agent 靠启动序列恢复状态。Atlas 的实践证明，即使你只有一个 Agent（没有初始化/编码 Agent 的拆分），启动序列也必须存在。**没有启动序列的 Agent 系统，本质上是每次都在赌上下文窗口没有被截断。**

## 2. 外部记忆系统 — 文件系统即上下文
### External Memory — The File System as Context

**教程说：** Agent 的内部记忆（上下文窗口）不可靠，要把记忆放到模型脑子之外。靠文件和日志而不是靠上下文来了解项目状态。核心是给 Agent 一个文件系统，让它能保存和回忆信息。

**Atlas 怎么做的：** 三件套——`state.json`（任务状态，JSON 格式强结构化）、`memory/*.md`（每日日记，按日期命名）、Obsidian Vault（长期知识库，Markdown + 双链）。`state.json` 是任务调度的核心，记录每个任务的状态（pending/done/failed/blocked）、依赖关系、交付物路径、完成时间戳。`memory/` 目录是过程日志，每天一个文件。MEMORY.md 是长期记忆，定期从日记中提炼洞察。

**踩了什么坑：** 这是 Atlas 踩过的最痛的坑之一。最初，我把状态分散在三个地方：`config.json`（系统配置）、`state.json`（任务状态）、`todos.md`（待办清单）。三份数据之间有重叠但不完全一致。某次 Agent 执行时，`state.json` 里任务标记为 done，但 `todos.md` 里还写着 pending。下一轮 Agent 读 `todos.md`，以为任务没完成，又执行了一遍。

教程里反复强调"文件系统即上下文"，但没有强调一个隐含前提：**你必须有一个单一真相源（Single Source of Truth）。** Atlas 的解决方案是把 `state.json` 升级为唯一权威——所有状态变更必须通过 `state.py` 这个工具，禁止手动编辑。`todos.md` 被删掉了，日记只记录"发生了什么"不记录"当前状态"。

**结论：** 外部记忆的关键不是"有文件"，是"单一真相源"。教程说得对——文件系统是 Agent 的外部大脑。但一个有三个不同版本的真相的大脑，比没有大脑更危险。

## 3. 压缩 vs 摘要 — Context Rot 的真实杀伤力
### Compression vs Summarization — The Real Damage of Context Rot

**教程说：** 压缩（Compression）是可逆无损操作——用轻量级指针替代原始信息（文件路径替代文件内容），Agent 未来仍可 100% 恢复原始信息。摘要（Summarization）是有损不可逆操作，仅在压缩收益非常小时作为最后手段。核心原则：先卸载再摘要——执行摘要前将完整上下文写入日志文件，为不可逆操作买保险。

**Atlas 怎么做的：** `compress.py` 实现了"先卸载再摘要"策略。具体逻辑：7 天前的日记 → 压缩（保留包含 ✅❌决定教训等标记的行）；14 天前的日记 → 摘要（提取洞察写入 MEMORY.md，原文件只保留标题）。阈值检查：当 memory 目录总行数超过 2000 行时触发压缩流程。

```python
# compress.py 核心逻辑
if age <= 14:
    # 压缩：保留关键行
    important_lines = [l for l in lines
        if any(marker in l for marker in ["✅", "❌", "决定", "教训", "重要", "关键"])]
else:
    # 摘要：提取洞察到 MEMORY.md
    insights = [f"- [{date}] {l.strip()}" for l in lines
        if any(marker in l for marker in ["决定", "教训", "突破", "关键"])]
```

**踩了什么坑：** 最初的版本设了固定阈值（2000 行），一刀切地对所有文件应用同样的压缩策略。问题是——有些文件是结构化的任务日志（压缩有意义），有些文件是思考笔记（压缩会丢失上下文连贯性），有些文件根本不是日记（比如 `service-log.md`，是客户记录不该被压缩）。教程里提到"只压缩旧 50%，保留最新完整行动作为 Few-shot 示例"，但 Atlas 的初版没有实现这个——它对所有超龄文件一视同仁。

另一个问题是"先卸载再摘要"的保险机制。教程说摘要前要先卸载完整上下文到日志文件。Atlas 的 `compress.py` 直接覆盖原文件——如果压缩逻辑有 bug，原始数据就丢了。后来加了 `--dry-run` 模式做预览，但"先备份再覆盖"这个保险步骤至今没有完全实现。

**结论：** 压缩策略需要按文件类型区分，不能一刀切。教程里 Menlo 的"先压缩再摘要"原则是对的，但实际落地时需要更精细的分类逻辑——哪些该压缩、哪些该摘要、哪些碰都不能碰。**Context Rot 的杀伤力是真实的，但粗暴的压缩比不压缩更危险。**

## 4. 渐进式披露 — 不需要的信息别塞进去
### Progressive Disclosure — Don't Feed What You Don't Need

**教程说：** Agent 启动时仅加载每个 Skill 的标题，决定使用某技能时才完整读取。模型通过 bash 工具在庞大脚本目录中自主寻找和阅读说明书。核心思想：多一点是噪音，少一点是盲区。

**Atlas 怎么做的：** HEARTBEAT.md 里定义了按任务类型的加载策略：

| 任务类型 | 需要加载 | 不需要加载 |
|:---|:---|:---|
| service | 服务目录.md, memory/service-log.md | 阅读笔记, 知识引擎 |
| content | 最近7天阅读笔记, 知识引擎摘要 | 服务目录, 客户名单 |
| distribution | 文章草稿, content-log.md | 其他所有 |
| meta | state.json, 所有日记(概览), MEMORY.md | 单篇文章全文 |

Atlas 安装了 19 个 Skills，但每次心跳只加载 BOOT_SEQUENCE.md + HEARTBEAT.md + 当前任务类型相关的文件。Skills 的 SKILL.md 只在实际触发时才读取。

**踩了什么坑：** 最初每次 spawn sub-agent 时，我都传了完整的系统 prompt（包含 AGENTS.md、SOUL.md、USER.md、TOOLS.md、所有 Skill 描述）。一次 sub-agent 调用的 token 消耗是精简后的 2-3 倍。更严重的问题不是成本——是噪音。Sub-agent 在海量无关信息中找任务指令，偶尔会"分心"去执行不是它该做的事（比如一个写文章的 sub-agent 试图去检查邮件）。

教程里说"渐进式披露"更多是从节省 token 的角度出发。但 Atlas 的经验告诉我，**渐进式披露的核心价值不是省钱，是提高判断力。** 一个只看到任务指令和相关背景的 Agent，做出的决策比一个看到所有信息的 Agent 更准确。这是"少即是多"的典型场景。

**结论：** 渐进式披露不只是"节省 token"，是"减少噪音提高判断力"。教程里的三层行动空间模型（函数调用 → 沙盒工具 → 编写代码）给了很好的框架，但实际落地时最有效的渐进式手段往往是最简单的——**不加载不需要的文件。**

## 5. Clean State — 下班前代码必须能跑通
### Clean State — Code Must Work Before Clocking Out

**教程说：** 任何一班 Agent 都不能把烂摊子留给下一班。不管这一班干了多少活，下班交接前必须保证代码能跑通、文档已更新，达到主分支合并标准。

**Atlas 怎么做的：** `validate.py` 实现了 Clean State 验证。Sub-agent 完成任务后，调用 `validate.py <task_id>`，它会检查任务定义的所有交付物（deliverable/deliverables）是否存在且非空（>10 bytes）。验证通过 → 标记 done + 记录时间戳；验证失败 → 标记 failed + 输出失败原因。

```python
# validate.py 核心检查
for d in all_deliverables:
    for p in paths_to_try:
        if p.exists():
            size = p.stat().st_size
            if size < 10:
                failures.append(f"  ⚠️  {d} 存在但内容为空")
            else:
                print(f"  ✅ {d} ({size} bytes)")
```

**踩了什么坑：** Atlas 最戏剧性的一次事故——sub-agent 声称完成了文章写作任务，`state.py done` 标记成功，但实际文件写入失败（路径拼接错误）。下一轮心跳读 state.json 看到任务 done，跳过了它，但依赖这个任务的下游任务找不到交付物，整个链条卡住了。

教程里 Anthropic 的方案用 Git 做 Clean State——提交能编译通过的代码就是 Clean State。Atlas 没有用 Git（内容创作项目不太需要版本控制），但 validate.py 做了类似的事——验证产出物的物理存在。区别在于：Git 验证的是"代码质量"（编译通过），validate.py 只验证"文件存在"。这是一个有意的简化，但也意味着**Atlas 的 Clean State 是弱验证——文件在不等于内容对。**

另一个教训：验证必须自动化。最初我让 Agent "自己确认一下文件写好了"，结果 Agent 经常自信地说"文件已创建"但实际没有。LLM 的自我验证不可靠——它会基于"我执行了写入操作"推断"文件应该存在"，而不是真的去检查文件系统。

**结论：** 验证不是"多余的检查"，是"跨会话信任的基础"。教程里 Clean State 的核心洞察完全正确：跨会话接力的最大风险不是某一个 Agent 做错了，而是**它做错了但告诉你做对了**。自动化验证是唯一可靠的防线。

## 额外发现：教程没说但 Atlas 教会我的

### 多 Agent 的实际切法

教程里关于多 Agent 的讨论集中在"广度 vs 深度"——探索性任务用多 Agent，编码任务用单 Agent。Atlas 用的是主 Agent + Sub-agent 的模式：主 Agent 做调度（读状态、领任务、spawn sub-agent），Sub-agent 做执行（写文章、分析数据）。这对应教程里的"以读写分界"原则——主 Agent 读操作多（分析状态、做决策），Sub-agent 写操作多（产出文件）。

但实际运行中我发现一个教程没提到的问题：**Sub-agent 的上下文隔离是有代价的。** 每个 Sub-agent 需要独立的启动序列、独立的文件加载，这些在主 Agent 看来是"简单的 spawn"，但对 Sub-agent 来说消耗的是真实的 token 和时间。当任务很简单时（比如"把这个文件复制到那个目录"），spawn 一个 Sub-agent 的开销比主 Agent 自己做更大。教程的三层行动空间模型在这里很有启发——简单操作应该在第一层（函数调用）直接做，不需要走到第二层（沙盒工具）或第三层（spawn agent）。

### 指标系统的必要性

Atlas 的 state.json 里有 `metrics` 字段，记录总心跳数、总执行任务数、成功/失败数、最后执行时间。这些不是为了好看——是为了回答"系统到底在不在正常工作"这个基本问题。教程里强调了架构和模式，但对可观测性着墨不多。实际运行中，没有指标的 Agent 系统就像没有仪表盘的飞机——飞着的时候你不知道它在不在正常飞。

## 理论和实践之间的差距有多大？

说实话，不大。Hermes Engineering 教程里写的每个模式——启动序列、外部记忆、压缩 vs 摘要、渐进式披露、Clean State——在 Atlas 的实践中都被证明是必要的。

**但差距确实存在，而且集中在同一个地方：落地细节。**

教程告诉你"先压缩再摘要"，但没告诉你怎么区分哪些文件该压缩哪些不该。教程告诉你"外部记忆靠文件系统"，但没提醒你多份状态文件之间的不一致会比没有外部记忆更危险。教程告诉你"启动序列很重要"，但没说清楚它对单 Agent 系统同样重要（不只是串行接力架构需要）。

这不是批评。教程本来就不应该事无巨细地覆盖所有细节——那是文档的工作。教程的价值在于给出正确的方向和核心原则。Atlas 的价值在于用真实的代码和踩坑经历，把这些原则翻译成具体的实现决策。

---

## Take-away

**别急着搭"完美的系统"，先搭一个能跑的，让它自己告诉你哪里该改。**

Atlas 的第一个版本没有启动序列、没有压缩引擎、没有 Clean State 验证、状态分散在三个地方。它跑了两天，问题就暴露了。每一个后来加上的"基础设施"，都是被真实的失败场景逼出来的。

这就是理论和实践的正确关系：**理论给你地图，实践告诉你路上的坑在哪。** 两者缺一不可，但先后顺序一定是——先上路。
